# phase-6 — MCP 구현 체크리스트

이 문서는 spec.md §5(완료 조건)와 analysis.md(구조·데이터 흐름·인터페이스·Decision Points D1~D8)를 실행 단위로
옮긴 순수 체크리스트다. 설계 근거는 analysis.md에 있다. 순서는 의존(다음이 가능하려면 무엇이 먼저 존재해야
하는가)을 따른다. 단일 클라이언트·어댑터·멀티서버의 동작 검증은 모두 같은 패키지 서버에 붙는 루프백으로만
가능하므로, 서버 노출 기반을 먼저 둔다.

## Section: 패키지 골격·서버

- [x] task-001: 서버 노출과 도구·프롬프트 등록
  - 목적: 이름을 지정해 MCP 서버를 만들고, 기존 도구 하나·도구 모음(레지스트리)·정적 프롬프트를 등록하면, 등록한
    도구가 이름·설명·입력 스키마를 갖춰 노출되고 호출 시 그 도구의 실행 결과가 텍스트와 오류 플래그로 돌아오며,
    등록한 프롬프트가 역할별 메시지로 노출된다. stdio 전송으로 서빙을 시작할 수 있다.
  - 접근: 신규 mcp 패키지에 SDK를 비공개 필드로 감춘 서버 래퍼와 전송·설정 타입을 두고, 우리 도구 스키마를 SDK
    입력 스키마로 역변환하며 우리 도구 실행을 호출하는 핸들러를 합성해 등록한다. 도구 실행에는 상태·스토어 없는
    최소 런타임을 넘긴다. SDK API(스키마 빌더·핸들러 시그니처·서빙 메서드)와 require 버전은 go get·go doc·컴파일로
    확인해 맞춘다.
  - 검증 조건:
    - 결과: 도구·도구 모음·프롬프트를 등록한 서버를 같은 패키지 전송으로 띄워 목록을 조회하면 등록한 도구가
      스키마와 함께 보이고, 호출하면 도구 실행 결과의 텍스트·오류 플래그가 그대로 돌아오며, 프롬프트 조회 시
      역할별 메시지가 돌아온다.
    - 확인: 등록·노출·호출을 단정하는 단위 테스트가 외부 프로세스·네트워크 없이 통과하고, `go build ./...`·
      `go vet ./...`가 오류 없이 끝난다.
  - 참조: SPEC §5.5, SPEC §5.1 / ANALYSIS §1, ANALYSIS §2, ANALYSIS §3, ANALYSIS §5 D4, ANALYSIS §5 D7

## Section: 클라이언트·어댑터

- [x] task-002: 단일 클라이언트 연결·목록·호출·프롬프트·종료
  - 목적: 설정을 받아 MCP 서버에 연결·초기화한 뒤, 원격 도구 목록을 우리 스키마 목록으로 받고, 이름과 인자로 도구를
    호출해 실행 결과를 우리 결과 타입으로 받으며, 프롬프트를 이름과 인자로 가져와 역할별 우리 메시지 목록으로 받고,
    연결을 정리한다.
  - 접근: SDK 클라이언트·세션을 비공개로 감춘 래퍼를 두고, 연결은 설정의 전송 종류에 따라 전송을 구성해 세션을 얻으며,
    목록·호출·프롬프트는 SDK 결과를 우리 스키마·결과·메시지로 변환한다. 입력 스키마 프로퍼티는 평면으로 순회해
    매개변수로 옮기고, 결과 텍스트 콘텐츠는 이어 붙이며 오류 플래그를 옮기고, 프롬프트 역할은 user/assistant로
    대응한다. SDK API는 go doc로 확인해 맞춘다.
  - 검증 조건:
    - 결과: 같은 패키지 서버에 클라이언트를 붙여 목록을 조회하면 원격 도구가 우리 스키마 목록으로 오고, 호출하면
      실행 결과가 우리 결과 타입(텍스트·오류 플래그)으로, 프롬프트는 역할별 우리 메시지 목록으로 오며, 종료가
      에러 없이 끝난다.
    - 확인: 연결·목록·호출·프롬프트·종료 왕복을 단정하는 단위 테스트가 외부 의존 없이 통과한다.
  - 참조: SPEC §5.2, SPEC §5.1 / ANALYSIS §1, ANALYSIS §2, ANALYSIS §3, ANALYSIS §5 D1, ANALYSIS §5 D2, ANALYSIS §5 D3

- [x] task-003: 원격 도구·프롬프트 어댑터
  - 목적: 원격 도구 목록을 우리 도구 인터페이스 구현 목록으로 받아 그 도구의 실행이 원격 호출로 결과를 내게 하고,
    단일 원격 스키마를 우리 도구로 감싸며, 원격 프롬프트를 우리 메시지 목록으로 받는다. 감싼 도구의 실행은 주입된
    런타임을 쓰지 않고 원격 호출에 위임한다.
  - 접근: 클라이언트의 목록 결과 각 스키마를 우리 도구 인터페이스 구현으로 감싸는 어댑터를 두고, 그 구현의 실행은
    입력을 그대로 원격 호출에 넘겨 받은 결과를 반환한다. 프롬프트 어댑터는 클라이언트 프롬프트 조회에 위임한다.
  - 검증 조건:
    - 결과: 같은 패키지 서버에 붙은 클라이언트에서 어댑터로 도구 목록을 받아 한 도구를 실행하면 원격 실행 결과가
      도구 출력에 반영되고, 단일 스키마 래핑이 우리 도구로 동작하며, 프롬프트 어댑터가 메시지 목록을 반환한다.
    - 확인: 어댑터 도구의 원격 실행·단일 래핑·프롬프트 변환을 단정하는 단위 테스트가 통과한다.
  - 참조: SPEC §5.3 / ANALYSIS §2, ANALYSIS §3, ANALYSIS §5 D2

- [x] task-004: 멀티서버 통합·서버별 조회·미등록 구분
  - 목적: 여러 서버 설정을 묶어 클라이언트를 만들고, 전체 도구를 통합 목록으로, 특정 서버의 도구만 따로, 특정
    서버의 프롬프트를 가져오며, 등록되지 않은 서버 이름으로 조회하면 not-found로 구분된다.
  - 접근: 서버 이름→단일 클라이언트 맵을 등록 순서를 보존해 구성하고, 통합 조회는 각 클라이언트 도구를 순서대로
    이어 붙이며, 서버별 조회·프롬프트는 해당 클라이언트에 위임하고, 미등록 이름은 판정 가능한 전용 에러로 돌려준다.
  - 검증 조건:
    - 결과: 두 개 이상의 같은 패키지 서버를 묶어 통합 조회하면 모든 서버 도구가 등록 순서로 합쳐지고, 서버별 조회는
      해당 서버 도구만, 프롬프트는 해당 서버 것만 돌아오며, 없는 서버 이름은 not-found 에러로 구분된다.
    - 확인: 통합·서버별·프롬프트·미등록 분기를 단정하는 단위 테스트가 외부 의존 없이 통과한다.
  - 참조: SPEC §5.4 / ANALYSIS §2, ANALYSIS §3, ANALYSIS §5 D6

## Section: 루프백·경계 검증

- [x] task-005: 인패키지 루프백 e2e
  - 목적: 이 패키지의 서버가 도구·프롬프트를 노출하고 같은 패키지의 클라이언트가 접속해 목록 조회→도구 호출→프롬프트
    로딩 왕복이 기대 결과를 내며, 외부 프로세스·네트워크·키 없이 결정적으로 실행되고 건너뛰지 않는다.
  - 접근: 같은 프로세스에서 서버를 띄우고 인메모리 전송으로 클라이언트를 물려 왕복을 검증하는 주 경로와, stdio
    전송 코드 경로 자체를 한 번 태우는 실증 경로를 함께 둔다. 가용성 가드(skip)는 두지 않는다. stdio 실증에 쓸
    SDK 전송 형태는 go doc로 확인해 맞춘다.
  - 검증 조건:
    - 결과: 루프백에서 목록 조회가 등록 도구를, 호출이 기대 실행 결과를, 프롬프트 로딩이 기대 메시지를 내고 왕복이
      통과하며, 어떤 환경에서도 skip 없이 실행된다.
    - 확인: 인메모리·stdio 두 경로의 왕복 e2e가 결정적으로 통과한다.
  - 참조: SPEC §5.6 / ANALYSIS §2, ANALYSIS §5 D5

- [x] task-006: import 경계 회귀 테스트
  - 목적: mcp 패키지가 도구·메시지·core(및 MCP SDK)만 import하고 상위 패키지(에이전트·그래프)를 import하지 않으며,
    하위 패키지가 mcp를 역참조하지 않음을 회귀로 보장한다.
  - 접근: import 그래프를 조회해 mcp의 의존 집합에 도구·메시지가 포함되고 금지 상위 패키지가 없는지, 하위 패키지가
    mcp를 import하지 않는지 검사하는 테스트를 작성한다.
  - 검증 조건:
    - 결과: mcp의 import에 도구·메시지·core가 있고 에이전트·그래프 등 상위가 없으며, 하위 패키지가 mcp를 import하지
      않는다.
    - 확인: import 경계 테스트가 통과하고(import 그래프 조회 기반), Phase 1~5 패키지 코드가 수정되지 않은 상태로
      모듈 전체 빌드·정적검사가 통과한다.
  - 참조: SPEC §5.7, SPEC §5.1 / ANALYSIS §4, ANALYSIS §5 D8
