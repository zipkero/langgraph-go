# a2a — A2A 프로토콜 구현 체크리스트

이 문서는 spec.md §5(완료 조건)와 analysis.md(구조·데이터 흐름·인터페이스·Decision Points D5.1~D5.4)를 실행 단위로
옮긴 순수 체크리스트다. 설계 근거는 analysis.md에 있다. 순서는 의존(다음이 가능하려면 무엇이 먼저 존재해야
하는가)을 따른다. 서버·클라이언트·어댑터·루프백의 동작 검증은 모두 와이어 타입과 서버 노출 기반을 전제하므로,
프로토콜 타입·직렬화를 먼저 두고 서버, 클라이언트, 어댑터, 루프백, 경계 순으로 쌓는다.

## Section: 프로토콜 타입·직렬화

- [x] task-001: 프로토콜 타입과 kind 판별 JSON 직렬화
  - 목적: 카드·스킬·능력, 태스크·태스크 상태·상태값, 메시지·파트(텍스트/데이터/파일)·아티팩트, 메시지 전송 파라미터·
    요청, 스트리밍 이벤트(상태 갱신·아티팩트 갱신·이벤트 union) 타입을 정의하고, 메시지·파트·파일 변형을 종류 문자열로
    판별해 직렬화·역직렬화하면 직렬화 후 역직렬화한 값이 원본과 같다. 상태값은 working/input_required/completed/
    failed/canceled 문자열로, 필드명은 camelCase 와이어 표현으로 나간다.
  - 접근: a2a 고유 타입을 정의하고, 파트·파일 union 래퍼에 종류 필드를 부착·판별하는 커스텀 직렬화를 둔다. 와이어
    필드명·종류 값은 README §22 명세에 맞춘다.
  - 검증 조건:
    - 결과: 텍스트/데이터/파일 파트를 가진 메시지·태스크·아티팩트를 직렬화한 뒤 역직렬화하면 종류 판별과 모든 필드
      값이 원본 그대로 보존되고, 상태값·필드명이 명세 문자열로 나간다.
    - 확인: 각 타입의 round-trip 보존을 단정하는 단위 테스트가 통과하고, `go build ./...`·`go vet ./...`가 오류 없이
      끝난다.
  - 참조: SPEC §5.2, SPEC §5.1 / ANALYSIS §1, ANALYSIS §3.1, ANALYSIS §5 D5.1

## Section: 서버

- [x] task-002: 서버 구성·카드 노출·태스크 수명주기·디스패치
  - 목적: 실행기·태스크 저장소를 받아 요청 핸들러와 서버를 구성해 띄우면 well-known 경로에서 카드가 조회되고, 메시지
    전송 요청이 실행기를 호출해 상태·아티팩트를 담은 태스크를 응답으로 돌려준다. 실행기 안에서 상태 갱신기로 working/
    input_required/completed/failed/canceled 전이·아티팩트 추가·완료·취소를 호출하면 그 전이와 아티팩트가 태스크
    상태와 스트리밍 이벤트(상태 갱신·아티팩트 갱신)에 반영되고, 최종 상태 이후 스트림이 종료된다. 태스크를 저장소에
    저장·조회할 수 있고 동시 접근에서 안전하다.
  - 접근: 이벤트 큐는 버퍼드 채널로, 상태 갱신기는 큐 push와 저장소 갱신을 한 호출에서 함께 수행하게 두고, 인메모리
    저장소는 잠금+맵으로 보호한다. 비스트리밍은 큐 이벤트를 최종 태스크로 수합, 스트리밍은 이벤트별 SSE 프레임으로
    흘린다. JSON-RPC 메서드 문자열은 message/send·message/stream으로 두되 외부 상호운용을 노릴 경우 스펙의
    바인딩 문자열을 1회 재확인한다. 재사용 심볼 시그니처는 컴파일로 맞춘다.
  - 검증 조건:
    - 결과: 테스트 서버를 띄워 well-known 경로를 직접 호출하면 카드가 오고, 메시지 전송 요청을 직접 호출하면 실행기가
      호출돼 상태·아티팩트를 담은 태스크가 응답으로 오며, 스트리밍 요청에서는 상태 갱신·아티팩트 갱신 이벤트가 최종
      상태 이벤트로 종료될 때까지 순차로 흘러나오고, 저장소 저장·조회가 동시 접근에서 일관되게 동작한다.
    - 확인: 카드 노출·디스패치·수명주기 전이·스트리밍 종료·저장소 동시성을 테스트 HTTP 서버로 직접 호출해 단정하는
      단위 테스트가 외부 네트워크 없이 통과하고, `go build ./...`·`go vet ./...`가 오류 없이 끝난다.
  - 참조: SPEC §5.3, SPEC §5.4, SPEC §5.1 / ANALYSIS §2.1, ANALYSIS §2.2, ANALYSIS §3.2, ANALYSIS §5 D5.2, ANALYSIS §5 D5.3

## Section: 클라이언트

- [x] task-003: 카드 조회·메시지 전송·스트리밍·아티팩트 추출
  - 목적: 카드 조회기가 well-known 경로에서 카드를 가져오고, 클라이언트의 메시지 전송이 원격 서버를 호출해 태스크를
    반환하며, 스트리밍 전송이 이벤트 채널을 반환해 상태 갱신·아티팩트 갱신·태스크 이벤트를 순차로 방출하고, 아티팩트
    추출 헬퍼가 아티팩트에서 텍스트·데이터·파일(URI·bytes)을 꺼낸다.
  - 접근: 메시지 전송은 요청을 JSON-RPC envelope로 POST해 result를 태스크로 역직렬화하고, 스트리밍은 SSE 응답을 라인
    단위로 소비해 각 data 프레임을 이벤트 union으로 디코드해 채널로 방출한다. 메서드 문자열·와이어 키는 서버와 같은
    상수를 공유한다.
  - 검증 조건:
    - 결과: 같은 패키지 서버에 클라이언트를 붙여 카드를 조회하면 카드가 오고, 메시지를 전송하면 상태·아티팩트를 담은
      태스크가 오며, 스트리밍 전송에서 이벤트가 채널로 순차 방출되고, 아티팩트 헬퍼가 텍스트·데이터·파일 URI·파일
      bytes를 각각 정확히 꺼낸다.
    - 확인: 카드 조회·전송·스트리밍·아티팩트 추출 왕복을 테스트 HTTP 서버에 붙여 단정하는 단위 테스트가 외부 의존
      없이 통과한다.
  - 참조: SPEC §5.5, SPEC §5.1 / ANALYSIS §2.3, ANALYSIS §3.3

## Section: 어댑터

- [x] task-004: 에이전트 스트림→태스크 수명주기 매핑
  - 목적: 에이전트 실행 스트림을 소비해 진행 중 이벤트는 working으로, 추가 입력 필요는 input_required(final)로, 완료는
    아티팩트 추가 후 완료로, 실행기 예외는 failed로, 취소는 canceled로 매핑한다. 기본 경로는 진행/추가입력/완료 플래그
    매핑이고, 구조화 응답이 에이전트 상태로 단언될 때만 대체 경로로 그 상태(input_required/completed/error)를 매핑하되
    error는 input_required로 흐른다.
  - 접근: 질의를 사용자 메시지로 감싼 입력과 세션 식별자를 실행 설정으로 전달해 에이전트 스트림을 호출하고, 방출되는
    이벤트를 루프 소비해 상태 갱신기로 전이·아티팩트를 매핑한다. 재사용하는 에이전트·구조화·메시지·설정 심볼
    시그니처는 컴파일로 맞춘다.
  - 검증 조건:
    - 결과: stub 또는 실제 에이전트 스트림을 어댑터에 흘리면 진행→working, 추가 입력 필요→input_required(final),
      완료→아티팩트 추가+완료, 예외→failed, 취소→canceled 전이가 일어나고, 구조화 응답 단언 시 대체 경로로 상태가
      매핑되며 error가 input_required로 흐른다.
    - 확인: 기본 경로 각 전이와 대체 경로(error→input_required 포함)를 단정하는 단위 테스트가 외부 의존 없이 통과한다.
  - 참조: SPEC §5.6 / ANALYSIS §2.4, ANALYSIS §3.3, ANALYSIS §5 D5.4

## Section: 루프백·경계 검증

- [x] task-005: 인패키지 루프백 e2e
  - 목적: 같은 패키지의 서버를 띄우고 같은 패키지의 클라이언트가 접속해 카드 조회→메시지 전송(및 스트리밍)→태스크·
    아티팩트 수신 왕복이 기대 결과를 내며, 외부 네트워크·원격 에이전트·키 없이 결정적으로 실행되고 건너뛰지 않는다.
  - 접근: 테스트 HTTP 서버로 서버를 기동하고 같은 패키지 클라이언트를 붙여 카드 조회·비스트리밍·스트리밍 왕복을
    검증한다. 가용성 가드(skip)는 두지 않는다.
  - 검증 조건:
    - 결과: 루프백에서 카드 조회가 카드를, 메시지 전송이 상태·아티팩트를 담은 태스크를, 스트리밍이 이벤트 시퀀스를
      내고 왕복이 통과하며, 어떤 환경에서도 skip 없이 실행된다.
    - 확인: 비스트리밍·스트리밍 두 경로의 왕복 e2e가 결정적으로 통과한다.
  - 참조: SPEC §5.7, SPEC §5.1 / ANALYSIS §2.1, ANALYSIS §2.2, ANALYSIS §2.3, ANALYSIS §4

- [x] task-006: import 경계 회귀 테스트
  - 목적: a2a 패키지가 agent·config·structured·message·tool·core와 표준 라이브러리만 import하고, gRPC/protobuf·외부
    a2a SDK가 의존 그래프에 없으며, 하위 패키지(특히 agent)가 a2a를 역참조하지 않음을 회귀로 보장한다. Phase 0~6
    패키지의 기존 동작은 수정되지 않는다.
  - 접근: import 그래프를 조회해 a2a의 의존 집합에 허용 패키지만 있고 금지 의존(gRPC/protobuf·외부 a2a SDK)이 없는지,
    하위 패키지가 a2a를 import하지 않는지 검사하는 테스트를 작성한다.
  - 검증 조건:
    - 결과: a2a의 import에 허용 패키지만 있고 금지 의존이 없으며, 하위 패키지가 a2a를 import하지 않는다.
    - 확인: import 경계 테스트가 통과하고, Phase 0~6 패키지 코드가 수정되지 않은 상태로 모듈 전체 빌드·정적검사가
      오류 없이 끝난다.
  - 참조: SPEC §5.8, SPEC §5.1 / ANALYSIS §3.4, ANALYSIS §4
