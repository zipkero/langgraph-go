# phase-5 — 장기 메모리 스토어 구현 체크리스트

이 문서는 spec.md §5(완료 조건)와 analysis.md(구조·Decision Points D1~D7)를 실행 단위로 옮긴 순수 체크리스트다.
설계 근거는 analysis.md에 있다. 순서는 의존(다음이 가능하려면 무엇이 먼저 존재해야 하는가)을 따른다.

## Section: store 코어

- [x] task-001: 네임스페이스 키-값 저장·조회·삭제
  - 목적: 네임스페이스로 분리된 영역에 맵 값을 저장하고, 같은 네임스페이스·키로 조회하면 동일 값을 돌려주며,
    삭제 후에는 조회되지 않고, 없는 키 조회는 not-found로 구분한다. 서로 다른 네임스페이스에 같은 키로 저장한
    값은 서로 독립적으로 보관·조회된다.
  - 접근: Namespace/Item/IndexConfig/Store 인터페이스와 동시성 안전한 인메모리 구현체, 옵션 생성자를 갖춘 신규
    패키지를 만든다. 네임스페이스→키→레코드 2단 맵을 sync.RWMutex로 보호하고, 반환 값은 복사본으로 돌려준다.
  - 검증 조건:
    - 결과: Put 후 같은 키 Get은 (저장값, true, nil), 없는 키 Get은 (nil, false, nil), Delete 후 Get은
      not-found, 동일 키를 다른 네임스페이스에 저장하면 각각 독립 조회된다.
    - 확인: stub EmbeddingClient 없이 결정적으로 도는 단위 테스트(저장/조회/삭제/네임스페이스 분리)가 통과하고,
      `go build ./...`·`go vet ./...`가 오류 없이 끝난다.
  - 참조: SPEC §5.2, SPEC §5.1 / ANALYSIS §1, ANALYSIS §2, ANALYSIS §5 D6

- [x] task-002: Item 기반 메타데이터 접근자
  - 목적: 저장 항목의 메타데이터(점수·생성 시각·수정 시각)를 Item으로 관찰할 수 있게 한다. 점수 동반 단건
    조회는 저장 시각 정보를 포함한 Item을 돌려주고, 비검색 경로의 점수는 0이다.
  - 접근: 단건 조회용 Item 반환 접근자를 추가하고, Put 시점에 신규는 생성 시각, 갱신은 수정 시각을 채워
    레코드에 보관한다.
  - 검증 조건:
    - 결과: 저장한 항목을 Item 접근자로 조회하면 값·네임스페이스·키·타임스탬프가 채워지고 점수는 0이며, 동일
      키 재저장 시 수정 시각이 갱신된다.
    - 확인: 메타데이터 필드와 점수 0을 단정하는 단위 테스트가 통과한다.
  - 참조: SPEC §5.2 / ANALYSIS §1, ANALYSIS §2, ANALYSIS §5 D2

- [x] task-003: 시맨틱 검색 정렬·limit·점수
  - 목적: 임베딩 인덱스를 설정하면 검색이 질의 임베딩과 저장 항목 임베딩의 코사인 유사도 내림차순으로 결과를
    정렬하고, 요청 개수만큼만 반환하며, 점수 동반 접근자로 각 결과의 유사도 점수를 확인할 수 있다.
  - 접근: 인덱스 설정 시 Put이 값의 텍스트 표현을 임베딩해 레코드에 보관하고, 검색은 질의를 임베딩해 자체 코사인
    유사도로 정렬·절단한다. 코사인은 외부 벡터 패키지를 빌리지 않고 패키지 안에 두며, 두 벡터의 더 짧은 길이까지
    합산해 길이 불일치에 견딘다.
  - 검증 조건:
    - 결과: 결정적 stub 임베딩으로 의미가 다른 항목들을 저장하고 한 항목에 가까운 질의로 검색하면 그 항목이
      1순위로 오고, limit만큼만 반환되며, 점수 동반 접근자 결과의 점수가 유사도 내림차순이다.
    - 확인: 유사도 정렬·limit 절단·점수 단정 단위 테스트가 stub EmbeddingClient로 네트워크 없이 통과한다.
  - 참조: SPEC §5.3 / ANALYSIS §2, ANALYSIS §5 D2, ANALYSIS §5 D3, ANALYSIS §5 D7

- [x] task-004: 인덱스 미설정 폴백 검색
  - 목적: 임베딩 인덱스를 설정하지 않은 상태에서 검색을 호출해도 에러나 패닉 없이, 해당 네임스페이스의 항목을
    키 정렬 순서로 요청 개수까지 반환하며 점수는 0이다.
  - 접근: 인덱스 미설정 분기를 검색 경로에 두어 질의를 무시하고 키 정렬·limit 절단한 결과를 점수 0으로 돌려준다.
  - 검증 조건:
    - 결과: 인덱스 없는 store에 여러 항목을 저장하고 검색하면 키 정렬 순서로 limit개가 반환되고 에러는 nil이며
      점수는 0이다.
    - 확인: 인덱스 미설정 폴백의 결정적 순서·개수·무에러를 단정하는 단위 테스트가 통과한다.
  - 참조: SPEC §5.3 / ANALYSIS §2, ANALYSIS §5 D4

- [x] task-005: context 주입·회수 헬퍼와 사용자 식별자 헬퍼
  - 목적: store를 context에 실어 두고 회수할 수 있게 하며, 회수 시 store가 없으면 not-found로 구분한다. 실행
    설정에서 사용자 식별자를 꺼내는 헬퍼를 제공한다.
  - 접근: 비공개 context 키로 store를 싣고 꺼내는 한 쌍의 헬퍼와, 실행 설정의 사용자 식별자 조회 래퍼를 추가한다.
  - 검증 조건:
    - 결과: store를 실은 context에서 회수하면 그 store와 true, 안 실은 context에서는 (nil, false), 사용자 식별자
      헬퍼는 설정에 있으면 그 값을, 없으면 빈 문자열을 반환한다.
    - 확인: 주입/회수 왕복과 부재 시 not-found, 식별자 헬퍼의 유무 분기를 단정하는 단위 테스트가 통과한다.
  - 참조: SPEC §5.4 / ANALYSIS §2, ANALYSIS §5 D5

## Section: 연동·검증

- [x] task-006: agent WithStore 주입 연동
  - 목적: 인메모리 store를 agent의 기존 store 주입 경로로 주입하면, 도구 함수가 런타임에서 받은 store에
    Put/Get/Search를 수행하고 그 결과가 도구 출력에 반영된다.
  - 접근: 테스트에서 store 구현체를 agent.WithStore로 주입하고, store를 호출하는 도구 함수를 실행해 결과를
    확인한다. 이 주입이 컴파일·동작하는 것으로 store가 agent 주입 계약을 구조적으로 충족함이 검증된다.
  - 검증 조건:
    - 결과: agent에 store를 주입하고 도구를 실행하면 도구가 rt.Store()로 받은 store에 저장한 값이 이후
      조회·검색에서 보이고 도구 출력에 반영된다.
    - 확인: agent.WithStore에 store 구현체를 넘기는 통합 테스트가 컴파일되고 통과한다(이 컴파일이 tool.Store
      구조적 충족의 검증 지점이다).
  - 참조: SPEC §5.5 / ANALYSIS §2, ANALYSIS §3, ANALYSIS §5 D1

- [x] task-007: 실제 임베딩 e2e (skip guard)
  - 목적: 실제 임베딩 서비스로 인덱스를 설정해 의미가 다른 항목을 저장하고 한 항목에 의미적으로 가까운 질의로
    검색하면 그 항목이 상위로 정렬된다. 임베딩 서비스/모델이 없으면 테스트를 건너뛴다.
  - 접근: 실제 임베딩 클라이언트로 인덱스를 구성한 e2e 테스트를 작성하고, 임베딩 가용성 체크 후 미가용이면
    t.Skip한다.
  - 검증 조건:
    - 결과: 임베딩 서비스가 있을 때 e2e가 실행되어 의미적으로 가까운 항목이 상위로 오고, 없을 때는 skip된다.
    - 확인: 임베딩 미가용 환경에서 테스트가 skip으로 통과하고, 가용 환경에서는 상위 정렬 단정이 통과한다.
  - 참조: SPEC §5.6 / ANALYSIS §2, ANALYSIS §5 D7

- [x] task-008: import 경계 회귀 테스트
  - 목적: store 패키지가 상위·동급 패키지를 import하지 않고 하위 패키지만 import하며, 하위 패키지가 store를
    역참조하지 않음을 회귀로 보장한다.
  - 접근: import 그래프를 조회해 store의 의존 집합에 금지 패키지가 없는지, 하위 패키지의 의존 집합에 store가
    없는지 검사하는 테스트를 작성한다.
  - 검증 조건:
    - 결과: store의 import에 tool·agent·graph·vectorstore가 없고 llm·config 등 하위만 있으며, 하위 패키지가
      store를 import하지 않는다.
    - 확인: import 경계 테스트가 통과한다(go list -deps 기반). Phase 1~4 패키지 코드가 수정되지 않은 상태로
      모듈 전체 빌드·정적검사가 통과한다.
  - 참조: SPEC §5.7, SPEC §5.1 / ANALYSIS §4, ANALYSIS §5 D7
